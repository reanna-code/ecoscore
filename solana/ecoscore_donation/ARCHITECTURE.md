## Why Solana?

For a consumer app where users earn small rewards per action, transaction costs matter enormously:

| Factor | Ethereum | Solana | Impact |
|--------|----------|--------|--------|
| **Transaction Cost** | $0.50 - $50 | $0.00025 | Users earn ~$0.05 per scan; ETH fees would exceed rewards |
| **Speed** | 12-15 seconds | 400ms | Real-time feedback when scanning products |
| **Throughput** | ~15 TPS | ~65,000 TPS | Scale to millions of product scans |
| **Finality** | ~6 minutes | ~400ms | Instant confirmation for users |
| **Developer Experience** | Mature but complex | Anchor makes it accessible | Faster development |

**Bottom line:** Ethereum's gas fees would destroy the micro-reward economics. Solana's near-zero fees make the model viable.

### 3. Events â€” On-Chain Receipts

When we emit events, Solana stores them permanently in the transaction log:

```rust
emit!(BatchDisburseEvent {
    week_id: 202605,
    total_points_pledged: 10000,
    total_amount_disbursed: 500_000_000,  // 0.5 SOL
    disbursements: [...],
    timestamp: 1738339200,
});
```

This becomes a **permanent receipt** proving:
- Exact amounts sent
- Which NGOs received funds
- When it happened
- How user pledges were honored

### batch_disburse â€” The Core Instruction

```rust
pub fn batch_disburse(
    ctx: Context<BatchDisburse>,
    week_id: u64,                      // e.g., 202605 (week 5 of 2026)
    allocations: Vec<BatchAllocation>, // [{ngo, points_pledged}, ...]
) -> Result<()>
```

**What it does:**
1. Validates week hasn't been processed before
2. Verifies all NGOs are whitelisted and active
3. Calculates pro-rata if pledges exceed vault
4. Transfers to all NGOs in one atomic transaction
5. Emits `BatchDisburseEvent` with full details

**Accounts required:**
- `admin` â€” Must match config.admin
- `config` â€” Program config PDA
- `ngo_registry` â€” NGO whitelist PDA
- `escrow_vault` â€” Vault holding funds
- `remaining_accounts` â€” NGO wallets in same order as allocations

### Executing Weekly Batch (Backend/Admin)

```typescript
async function executeBatchDisburse(
  program: Program,
  admin: Keypair,
  weekId: number,
  allocations: { ngo: PublicKey; pointsPledged: number }[]
) {
  const tx = await program.methods
    .batchDisburse(
      new BN(weekId),
      allocations.map(a => ({
        ngo: a.ngo,
        pointsPledged: new BN(a.pointsPledged),
      }))
    )
    .accounts({
      admin: admin.publicKey,
      config: configPda,
      ngoRegistry: ngoRegistryPda,
      escrowVault: vaultPda,
      systemProgram: SystemProgram.programId,
    })
    .remainingAccounts(
      allocations.map(a => ({
        pubkey: a.ngo,
        isSigner: false,
        isWritable: true,
      }))
    )
    .signers([admin])
    .rpc();

  return tx;
}
```

## Security Model

### On-Chain Protections

| Protection | Implementation | Purpose |
|------------|----------------|---------|
| Admin-only operations | `constraint = admin.key() == config.admin` | Only admin can disburse |
| NGO whitelist | Registry check before disbursement | Prevent arbitrary recipients |
| Week deduplication | `require!(week_id > last_batch_week)` | Prevent double-spending |
| Amount validation | `require!(amount > 0)` | No zero-value transactions |
| Overflow protection | `checked_add()` everywhere | Prevent arithmetic exploits |
| PDA signatures | `invoke_signed` for transfers | Only program can move vault funds |
| Batch size limits | Max 10 NGOs per batch | Fit in transaction size |
